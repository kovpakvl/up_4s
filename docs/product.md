# SecureOffice: продуктовая документация

## Идея

SecureOffice — учебный менеджер корпоративных доступов для небольшой локальной сети. Системный администратор запускает desktop-приложение у себя, поднимает сервер и управляет структурой компании. Сотрудники подключаются через браузер и работают только со своими паролями.

## Ценность

Продукт закрывает учебный сценарий:

- один ответственный администратор;
- локальная сеть без внешнего облака;
- централизованное создание сотрудников и ключей доступа;
- раздельный доступ администратора и сотрудников;
- понятный журнал действий;
- web-кабинет без установки ПО сотрудниками.

## Роли

### Системный Администратор

Основной пользователь desktop-приложения.

Может:

- запускать и останавливать сервер;
- создавать первый аккаунт администратора;
- создавать отделы и должности;
- добавлять сотрудников;
- выдавать ключи активации;
- создавать записи паролей сотрудникам;
- смотреть и менять записи;
- смотреть журнал событий.

### Сотрудник

Пользователь browser-кабинета.

Может:

- активировать аккаунт по индивидуальному ключу;
- входить по логину и паролю;
- видеть свои записи паролей;
- добавлять и редактировать свои записи;
- копировать пароль;
- смотреть историю пароля.

## Основной Сценарий

1. Администратор запускает `SecureOfficeAdmin.exe`.
2. Нажимает `Запустить сервер`.
3. Создаёт первый аккаунт администратора.
4. Добавляет отделы и должности.
5. Добавляет сотрудников.
6. Выдаёт каждому сотруднику ключ.
7. Сотрудник активирует аккаунт в браузере.
8. Сотрудник работает со своими паролями.
9. Администратор при необходимости контролирует записи и журнал.

## MVP-Функции

В текущей версии есть:

- desktop-приложение администратора;
- backend на Flask;
- PostgreSQL через Docker Compose;
- web-кабинет сотрудника;
- активация сотрудника по ключу;
- отделы и должности;
- записи паролей;
- история паролей;
- журнал событий;
- onboarding для первого запуска;
- exe-сборка через PyInstaller.

## Границы MVP

В учебной версии осознанно не закрыты некоторые production-требования:

- нет HTTPS;
- нет доменной авторизации;
- нет отдельного backup/restore интерфейса;
- нет ротации master-ключа шифрования;
- нет сложной RBAC-матрицы на уровне UI;
- Docker Desktop устанавливается отдельно.

## Решения По Продукту

- Администратор работает в desktop-приложении, потому что ему нужно управлять локальным сервером.
- Сотрудники работают в браузере, потому что им не нужно устанавливать ПО.
- Ключ активации индивидуален и одноразов по смыслу сценария.
- Парольная запись принадлежит конкретному сотруднику.
- Администратор может создать одинаковую запись сразу нескольким сотрудникам.
- Все заметные действия пишутся в журнал.
- Время отображается по МСК.

## Следующие Итерации

Наиболее полезные дальнейшие шаги:

1. Backup/restore из desktop-приложения.
2. Настройка секретов при первом запуске вместо фиксированных значений compose.
3. HTTPS для web-кабинета.
4. Настройка срока действия ключей в UI.
5. Более подробные роли внутри админки.
6. Импорт/экспорт сотрудников.
7. Отдельный release workflow на GitHub.
8. Установка Docker Desktop через понятную ссылку/проверку в приложении.

## Критерии Готовности Учебного Проекта

Проект можно считать завершённым для учебной защиты, если:

- exe запускает desktop-приложение;
- администратор может поднять сервер;
- первая настройка проходит без ручного редактирования БД;
- сотрудник может активироваться по ключу;
- сотрудник видит только свои записи;
- администратор видит структуру, пароли и журнал;
- тесты проходят;
- в репозитории есть техническая, пользовательская и продуктовая документация.
